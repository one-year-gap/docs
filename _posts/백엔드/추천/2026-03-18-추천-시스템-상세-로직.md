## 통신 추천 시스템 아키텍처 개요

이 문서는 현재 구현된 \*\*LLM 기반 RAG 추천 시스템\*\*의 구조를 발표용으로 한눈에 정리한 문서입니다.
핵심 목표는 \*\*모바일 편향 완화\*\*, \*\*고객 컨텍스트 기반 개인화\*\*, \*\*안전한 룰 기반 필터링(연령·타입)\*\* 입니다.
    ---

## 1. 전체 흐름 한 줄 요약

- \*\*입력\*\*: \`memberId\` 로 추천 요청
- \*\*컨텍스트 로딩\*\*: \`member_llm_context\` + \`product\` 테이블
- \*\*Retrieval\*\*: 쿼리 임베딩 + pgvector 유사도 검색 (후보 50개)
- \*\*후보 재가공\*\*: 연령 태그 하드 필터 + 타입 분산 + TOP-K(3개) 보장
- \*\*LLM 단계\*\*: 세그먼트/페르소나별 프롬프트로 이유(reason) 생성
- \*\*최종 후처리\*\*: 타입별 최대 2개 제한, Fallback/에러 핸들링
- \*\*평가\*\*: \`scripts/test_recommendation_eval.py\` 로 Hit@k, Recall@k 측정 

## 2. 데이터 & 컨텍스트 구조

### 2.1 `member_llm_context`(고객 컨텍스트)

고객 한 명당 LLM이 이해할 수 있는 핵심 정보를 정규화해서 모아 둔 테이블입니다. 대표 컬럼은 다음과 같습니다.

    - \*\*기본 정보\*\*
    - \`member_id\`, \`membership\`, \`age_group\`, \`join_months\`, \`family_group_num\`, \`family_role\`
- \*\*세분화/페르소나\*\*
    - \`segment\`: \`CHURN_RISK\`, \`UPSELL\`, \`NORMAL\` 등
    - \`persona_code\`: \`SPACE_SHERLOCK\`, \`SPACE_SURFER\`, \`SPACE_GUARDIAN\` 등
- \*\*현재 이용/관심 신호\*\*
    - \`current_subscriptions\`: 현재 가입 상품 목록(JSON)
    - \`current_product_types\`: \`{ "MOBILE_PLAN": true, "INTERNET": false, ... }\`
    - \`product_type_clicks\`: \`{ "MOBILE_PLAN": 5, "INTERNET": 3, ... }\`
    - \`recent_viewed_tags_top_3\`: \`["데이터무제한", "가족결합메인", ...]\`
- \*\*이용 패턴 & 이탈 위험\*\*
    - \`current_data_usage_ratio\`, \`data_usage_pattern\`
    - \`churn_score\`, \`churn_tier\`
    - \`recent_counseling\`, \`contract_expiry_within_3m\`

이 정보로 “어떤 상품 유형에 관심이 크고, 어떤 라이프스타일/위험군인지”를 LLM과 벡터 검색 모두에서 활용합니다.

### 2.2 \`product\` (상품 마스터 + 임베딩)

    - \*\*기본 정보\*\*
    - \`product_id\`, \`product_name\`, \`product_type\`
    - \`price\`, \`sale_price\`, \`tags\` (예: \`["청소년", "데이터무제한", "OTT"]\`)
- \*\*벡터 검색 관련\*\*
    - \`embedding_text\`: 상품 설명 텍스트
    - \`embedding_vector\`: OpenAI 임베딩 → \`pgvector\` 컬럼
    ---

## 3. 추천 파이프라인 전체 구조

### 3.1 API 입력

    - 엔드포인트: \`POST /api/v1/recommendations\`
- 요청 바디: \`{"memberId": <int>}\`
- FastAPI 엔트리: \`app/realtime/api/v1/recommendation.py\`

### 3.2 컨텍스트 로딩

    - \`memberId\` 기준으로 \`member_llm_context\` 조회
    - \*\*있으면\*\*: 풀 컨텍스트 기반 LLM 추천 경로
    - \*\*없으면\*\*: 컨텍스트 없는 Fallback 경로 (기본 상품 + 간단 LLM/고정 메시지)

### 3.3 Retrieval 쿼리 텍스트 생성 (\`retrieval_query_builder.py\`)

\*\*목표\*\*: “모바일만 잘 나오는 임베딩”이 아니라, 고객이 실제로 관심 가질 만한 \*\*다양한 상품 유형\*\*이 벡터 검색 후보에 섞이도록 쿼리 텍스트를 설계합니다.

    - \`current_product_types\`, \`product_type_clicks\`, \`recent_viewed_tags_top_3\` 를 종합해서 \`interest_types\` 계산
    - 현재 사용 중인 타입에 높은 가중치
    - 클릭 수가 많은 타입에 가중치
    - 최근 본 태그(\`recent_viewed_tags_top_3\`)가 붙어 있는 상품 타입에 가중치
- \`TYPE_LABELS\` 를 사용해 한글 타입 라벨 생성
    - \`MOBILE_PLAN\` → “휴대폰 요금제”
    - \`INTERNET\` → “인터넷 상품”
    - \`IPTV\` → “IPTV 상품”
    - \`TAB_WATCH_PLAN\` → “태블릿/워치 요금제”
    - \`ADDON\` → “부가서비스”
- 쿼리 텍스트 마지막에 다음과 같은 문장을 추가:
    - \`특히 휴대폰 요금제, 인터넷 상품, IPTV 상품 중에서 고객에게 어울리는 다양한 상품 유형을 함께 고려해 주세요.\`

이로 인해 임베딩 단계부터 “모바일 외 타입도 같이 고려해 달라”는 힌트를 주게 됩니다.

### 3.4 임베딩 + 벡터 검색

    - OpenAI 임베딩으로 쿼리 벡터 생성
- \`pgvector\` 기반 유사도 검색으로 \*\*최대 50개(\`RETRIEVAL_CANDIDATES_K\`)\*\* 후보 탐색
- \`compute_product_type_weights(ctx)\` 를 통해 타입별 가중치 계산
    - \`SEGMENT_WEIGHT_CONFIG\` 기반으로 segment 별 가중치 튜닝
    - 예: 이탈 위험 세그먼트면 현재 상품 타입 유지 가중치를 더 크게, 업셀 세그먼트면 상위 스펙 타입 가중치를 더 크게 등
- \`_product_type_boost_from_weights(weights)\` 로 가중치를 score에 반영하여 \*\*모바일 편향 완화 + 고객 관심 타입 Boost\*\*

### 3.5 후보 재정렬 & 필터링 (\`recommendation_service.py\`)

#### 3.5.1 데이터 사용 패턴 기반 재정렬

    - \`_reorder_by_data_usage_pattern(products_ordered, ctx["data_usage_pattern"])\`
    - 데이터 과소/과다/패턴에 맞게 일부 상품 우선권 조정

#### 3.5.2 연령 태그 기반 하드 필터 (\`_is_age_allowed_for_tags\`)

    - \`_age_from_ctx(ctx)\` 로 \`age_group\` → 실제 나이 추정
- 상품 \`tags\` 에 다음과 같은 연령 관련 태그가 있는 경우, 고객 나이와 \*\*완전히 맞지 않으면 제외\*\*
    - \`"키즈"\`, \`"자녀보호"\`, \`"청소년"\`, \`"20대청년"\`, \`"시니어"\`, \`"현역병사"\`, \`"복지혜택"\` 등
- 이 필터는 \*\*하드 룰\*\*로, 추천 개수가 줄어들더라도 절대 완화하지 않습니다.

이를 통해 “성인에게 키즈 요금제 노출” 같은 실수를 원천 차단합니다.

#### 3.5.3 타입 분산 & TOP-K 보장 (\`_diversify_products_by_type\`)

    1. 연령 필터를 통과한 후보들(\`base_candidates\`)에 대해:

    - \`_diversify_products_by_type(base_candidates, max_per_type=MAX_PRODUCTS_PER_TYPE, max_total=5)\`
    - \`MAX_PRODUCTS_PER_TYPE = 2\` (파일 상단 상수)
    - 각 \`product_type\`별 최대 2개, 전체 최대 5개까지 1차 후보를 구성

2. 이 1차 후보가 \`top_k=3\` 이상이면 그대로 상위 3개 사용
3. \`top_k\`보다 적으면:

    - 이미 사용한 product_id를 제외하고, 연령 필터를 통과한 \`base_candidates\`에서 추가로 채워 넣음
    - \*\*타입 분산 규칙은 완화\*\*, 하지만 연령 필터는 그대로 유지

이 구조 덕분에:

    - \*\*최소 3개 추천\*\*을 최대한 보장하면서
- \*\*타입별로 골고루 섞인 후보\*\*를 LLM에 전달할 수 있습니다.

## 4. LLM 추천 단계

### 4.1 LLM 입력 구조

LLM에는 다음과 같은 정보가 JSON/텍스트로 전달됩니다.

    - \*\*고객 컨텍스트\*\*
    - 세그먼트(\`segment\`), 페르소나(\`persona_code\`)
    - \`current_product_types\`, \`product_type_clicks\`, \`recent_viewed_tags_top_3\`
    - \`data_usage_pattern\`, 최근 상담 내용(\`recent_counseling\`), 가족/결합 정보 등
- \*\*후보 상품 리스트\*\*
    - 앞 단계에서 타입 분산+연령 필터를 거친 3\~5개 상품
    - 각 상품에 대해 \`product_id\`, \`product_name\`, \`product_type\`, \`price\`, \`sale_price\`, \`tags\` 등

### 4.2 프롬프트 설계 (세그먼트 + 페르소나)

#### 4.2.1 세그먼트(system prompt) (\`persona_recommendation_prompts.py\`)

    - \`SEGMENT_SYSTEM_PROMPTS["CHURN_RISK" | "UPSELL" | "NORMAL"]\`
    - \*\*공통 규칙\*\*
        - “반드시 JSON으로만 응답”
        - 추천 상품은 최대 3개
        - 각 상품의 \`reason\`은 \*\*마케팅 카피 톤\*\*으로 2\~3문장
        - \*\*고객의 현재 상황과 비교 설명 필수\*\*
            - 예: 현재 요금제/지출 수준 대비 얼마나 절약/혜택 증가/안정성 향상이 있는지
            - 최근 상담 내용, 데이터 이용 패턴, 가족 구성 등을 최소 1번 이상 구체적으로 언급
        - “단순 카탈로그 설명”이나 “일반적인 홍보 문구”는 금지
- 세그먼트 별로 톤과 포커스만 다르게 설계
    - \`CHURN_RISK\`: 불안/불편 감소, 비용 부담 완화에 포커스
    - \`UPSELL\`: 혜택 확장, 라이프스타일 업그레이드
    - \`NORMAL\`: 균형적인 최적 조합, 가성비/편의성 중심

#### 4.2.2 페르소나 스타일(prompt) (\`PERSONA_STYLE_PROMPTS\`)

    - 예: \`SPACE_SHERLOCK\` (우주 셜록 홈즈)
    - 비용 효율, 절약, 최저가 조합에 집착하는 고객 페르소나
    - reason은 “얼마나 아낄 수 있는지”, “같은 가격에 어떤 혜택이 더 붙는지”를 숫자/비율로 설명
    - 문장 예시를 프롬프트 안에 포함시켜 톤과 길이를 고정
- 예: \`SPACE_GUARDIAN\` (세이프 가디언)
    - 가족/자녀 보호, 안정성, 안심 요금제에 민감한 페르소나
    - 자녀 데이터 차단, 위치 보호, 과금 폭탄 방지 같은 키워드를 강조
- 이 외 페르소나들도 각각 마케팅 카피 스타일과 비교 포인트를 명시

이렇게 세그먼트 + 페르소나 2축으로 \*\*LLM reason의 톤과 내용 구조\*\*를 제어합니다.

### 4.3 LLM 출력 형식 & 처리

    - LLM은 반드시 JSON으로 응답:
    - 예시: \`{"products": [{"productId": 1, "reason": "..."}, ...]}\`
- 서비스단에서는:
    - LLM이 선택한 productId 순서를 유지
    - reason이 비어 있는 경우 \`DEFAULT_PRODUCT_REASON_TEXT\` 로 대체
    - LLM이 반환하지 않은 productId는 후보군에서 보완
    ---

## 5. 최종 추천 조합 & 응답 스키마

### 5.1 타입별 최대 개수 제한 (\`MAX_PRODUCTS_PER_TYPE\`)

    - 실제 응답 조립 단계에서 \`MAX_PRODUCTS_PER_TYPE = 2\` 상수를 기준으로:
    - 한 \`product_type\` 이 \*\*최대 2개까지만\*\* 포함되도록 \`type_counts\` 딕셔너리로 제어
    - 과거의 “모바일 요금제만 3개 나오는” 문제를 구조적으로 차단
- LLM이 3개 모두 \`MOBILE_PLAN\` 을 골라도:
    - 최대 2개까지만 반영, 나머지는 다른 타입 후보로 교체

### 5.2 응답 스키마 (\`app/schemas/recommendation.py\`)

    - \*\*루트 응답\*\*: \`RecommendationResponse\`
    - \`segment\`: 고객 세그먼트
    - \`cached_llm_recommendation\`: LLM이 생성한 전체 추천 요약 문구
    - \`recommended_products\`: 추천 상품 배열 (최대 3개)
    - \`source\`: \`LIVE\` / \`FALLBACK\` 등
    - \`updated_at\`: ISO8601 타임스탬프
- \*\*상품 항목\*\*: \`RecommendedProductItem\`
    - \`product_id\`, \`product_name\`, \`product_type\`
    - \`product_price\`, \`sale_price\`, \`tags\`
    - \`llm_reason: str = Field(..., serialization_alias="reason")\`
        - 외부 JSON에서는 \*\*\`reason\`\*\* 필드 이름으로 노출
    ---

## 6. Fallback 전략

### 6.1 \`member_llm_context\` 가 없을 때

    - 해당 \`member_id\` 컨텍스트가 없는 경우:
    - 기본적인 쿼리 텍스트로 임베딩 + 검색
    - 혹은 사전에 정의된 인기 상품/기본 패키지 위주 추천

### 6.2 OpenAI/임베딩 오류 시 (\`_run_fallback_recommendation\`)

    - 임베딩 생성 실패, OpenAI API 에러 등 발생 시:
    - \`FETCH_DEFAULT_PRODUCTS_SQL\` 로 미리 정의된 기본 상품 목록 조회
    - \`FALLBACK_CACHED_MESSAGE\` / \`FALLBACK_VECTOR_SUMMARY\` 기반 안내 문구 사용
        - 예: \`"아직 추천을 생성할만한 정보가 없어요. 대신 이 상품은 어떠세요?"\`
    - 이때도 \`MAX_PRODUCTS_PER_TYPE = 2\` 규칙을 적용하여 타입 편향 방지

### 6.3 조건 미충족 케이스

    - 연령 태그 필터 등으로 인해 추천 가능한 상품이 아예 없으면:
    - \`NO_MATCHED_MESSAGE = "조건에 맞는 추천 상품이 없습니다."\`
    - \`recommended_products\` 는 빈 배열로 반환
    ---

## 7. 오프라인 평가 구조 (\`scripts/test_recommendation_eval.py\`)

### 7.1 입력 CSV 형식

    - 필수 컬럼:
    - \`member_id\`: 테스트할 회원 ID
    - \`recommended_products\` \*\*또는\*\* \`expected_products\`:
        - \`"1,2,3"\` 또는 \`"[1, 2, 3]"\` 형태의 product_id 리스트

### 7.2 평가 로직

    - 각 row 에 대해:

    1. \`get_recommendation(session=None, member_id=member_id)\` 호출
    2. 응답의 \`recommended_products[:k]\` → \`returned_ids\`
    3. 정답 집합 \`expected\` 와의 교집합으로:

        - \`hit@k = 1 if 교집합 존재 else 0\`
        - \`recall@k = |교집합| / |expected|\`
- 전체 케이스에 대해:
    - \`Hit@k = (hit@k 합계) / N\`
    - \`Recall@k = (recall@k 합계) / N\`

### 7.3 실행 예시

\`\`\`bash
python -m scripts.test_recommendation_eval \
    --csv scripts/final_dataset2.csv \
    --k 3 \
    --out-csv scripts/eval_results.csv
\`\`\`

    - \`--out-csv\` 지정 시, 각 케이스별로:
    - \`member_id\`, \`expected_products\`, \`returned_products\`, \`reason\`, \`hit@k\`, \`recall@k\`
    - 를 CSV로 저장하여 \*\*정성 평가(LLM reason 내용)\*\* 까지 함께 확인 가능
    ---

## 8. 설계 핵심 포인트 요약 (발표용 메시지)

    - \*\*모바일 편향 완화\*\*를 위해, 단일 단계가 아닌 \*\*다단계 제어\*\*를 설계했습니다.
    - 쿼리 임베딩 텍스트에서부터 다양한 상품 유형을 명시
    - pgvector 검색에서 타입 가중치로 랭킹 조정
    - 후보 단계에서 타입 분산(\`MAX_PRODUCTS_PER_TYPE = 2\`) 적용
    - 최종 응답에서도 다시 한 번 타입별 최대 개수 제한
- \*\*안전한 룰 기반 필터\*\*를 병행합니다.
    - 연령 태그(키즈/청소년/시니어 등)는 \*\*하드 필터\*\*로 적용하여 오·폭주 추천을 차단
    - 조건 미충족 시에는 깔끔하게 “조건에 맞는 추천 상품이 없습니다.” 메시지 반환
- \*\*LLM은 톤·구조를 강하게 제어\*\*합니다.
    - 세그먼트/페르소나별로 프롬프트를 분리하여, 마케팅 카피 스타일과 비교 포인트를 구체적으로 지시
    - “현재 요금제/지출 대비 어떤 점이 더 나은지”를 반드시 reason에 포함하도록 규칙화
- \*\*운영·평가 가능성\*\*을 고려했습니다.
    - \`scripts/test_recommendation_eval.py\` 로 Hit@k, Recall@k 를 자동 측정
    - CSV 단위로 정답/예측/LLM reason 까지 함께 로그로 남겨, 오프라인 튜닝과 개선 사이클을 돌릴 수 있습니다.
